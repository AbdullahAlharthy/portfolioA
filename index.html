Graduation Project · Machine Learning

Customer Churn Prediction

Built a predictive system to identify telecom customers at risk of churn — following CRISP-DM phases from data understanding through model evaluation and SHAP-based explainability.

PythonLightGBMTabDDPMSMOTETomekH2O AutoMLPyCaretSHAPOptunaScikit-learnPandas
View on GitHub
🎯 The Problem
Telecom companies lose significant revenue to customer churn. Acquiring a new customer costs 5× more than retaining an existing one — making early churn prediction a high-value business problem.
⚙️ Our Approach
We applied CRISP-DM methodology — benchmarking 9 ML algorithms across 6 balancing strategies and 4 encoding schemes over 10+ documented runs. Multiple imbalance techniques (SMOTE, SMOTEENN, class weights) were evaluated. The final pipeline: TabDDPM was trained on the training set only to generate synthetic minority samples, then SMOTETomek refined the boundary by removing noisy overlaps.
✅ Key Result
Final pipeline: LightGBM + TabDDPM + SMOTETomek + Threshold Tuning — F1=0.67, Recall=0.78, Precision=0.58. The same pipeline evaluated with synthetic data (TabDDPM) outperformed the version without it by ~7% in F1 (0.63 → 0.67). SMOTETomek then refined the decision boundary. Recall was prioritized: missing a churner costs 5× more than a false positive.

Heart Disease Prediction

Analyzed 2,182 patient records (1,889 after cleaning) using RapidMiner. Three tasks: EDA, Decision Tree classification (12 models compared), and K-Means clustering for patient segmentation.

RapidMinerDecision TreeK-Means ClusteringEDAStatistical Analysis
View on GitHub
🎯 The Problem
Heart disease is a leading cause of death globally. Clinical datasets from different sources vary in format, scale, and completeness — making multi-source data integration a core challenge before any modeling can begin.
⚙️ Our Approach
We merged 5 public datasets (2,182 raw records) into one clean 1,889-record dataset with 14 clinical features. RapidMiner was used for its visual, reproducible pipeline design. Three analysis tasks: (1) EDA for feature understanding, (2) Decision Tree classification across 12 configurations, (3) K-Means for exploratory patient segmentation — to discover hidden patient subgroups, not for prediction.
✅ Key Result
Best decision tree: Information Gain + 3.5% min leaf — 85% accuracy, overall score 0.87, Lift=1.20. In this exploratory study, accuracy was the primary metric since the goal was model comparison across configurations, not deployment. K-Means (K=3 via Elbow Method) revealed three clinically distinct patient groups — a complementary unsupervised view of the same data.

Deep Learning · Object Detection

Object Detection in Drone Imagery

Trained YOLOv8 and YOLO11 models on VisDrone2019 — one of the hardest aerial detection benchmarks. Scaled data from 6,471 to 12,942 images via custom augmentation. Best: YOLO11l at 34.6% mAP@50 with 1.9ms real-time inference.

YOLO11lYOLOv8mYOLOv8nPyTorchCUDAOpenCVUltralyticsImage ProcessingData Augmentation
View on GitHub
🎯 The Problem
VisDrone2019 is one of the most challenging object detection benchmarks: objects are often under 10px in width, scenes contain hundreds of overlapping instances, and images span varying altitudes. Class imbalance is also severe — cars dominate while bicycles and awning-tricycles are rare, directly affecting per-class detection performance.
⚙️ Our Approach
We designed a custom augmentation pipeline (ColorJitter, HorizontalFlip ±50%, RandomRotation ±10°, HSV adjustments) to double training data from 6,471 to 12,942 images. Four experiments scaled both model size and data volume: train6 (YOLOv8n, 10 epochs) → train4 (YOLOv8n, 20 epochs) → train3 (YOLOv8m) → train7 (YOLO11l). Both train3 and train7 achieved mAP@50=0.346, but YOLO11l was selected for its superior architecture depth (190 layers, 25.3M params) and 1.9ms inference speed — making it deployable in real-time UAV systems.
✅ Key Result
YOLO11l achieved mAP@50 = 0.346 · mAP@50-95 = 0.210 · Inference: 1.9ms/image. Augmentation alone (train6→train4): mAP improved from 0.236 to 0.250 (+6%) with the same YOLOv8n — data volume was the primary bottleneck. Model scaling then pushed to 0.346. The moderate overall mAP reflects the inherent difficulty of small-object aerial detection, not a model limitation.
